マネージドサービス
一言でいうと
面倒な管理をクラウド事業者が代わりにやってくれるサービス。

具体例

  • サーバーの更新
  • 障害対応
  • バックアップ
  • OSのメンテナンス
例え
自分で車を所有するのではなく、カーシェアを利用するイメージ。
整備は運営会社が行い、自分は運転だけをすればよい。
講師メモ
クラウドは「全部自分で管理する」ではなく、 管理を任せられるところに価値があります。
IaaS（Infrastructure as a Service）（インフラストラクチャ・アズ・ア・サービス）
一言でいうと
サーバーだけ借りるサービス。

利用者が管理するもの

  • OS
  • ミドルウェア
  • アプリケーション

クラウド事業者が管理するもの

  • サーバー本体
  • ストレージ
  • ネットワーク
例
Amazon EC2
Azure Virtual Machines
例え
更地（土地）だけ借りるイメージ。
家は自分で建てる。
PaaS（Platform as a Service）（プラットフォーム・アズ・ア・サービス）
一言でいうと
プログラムを作ることに集中できるサービス。

OSやミドルウェアの準備はサービス側が行います。

例
  • Azure App Service
  • Google App Engine
例え 完成したキッチン付きのレンタル厨房。 料理だけ作ればよい。
SaaS（Software as a Service）（ソフトウェア・アズ・ア・サービス）
一言でいうと 完成したソフトをそのまま利用するサービス。
例
  • Microsoft 365
  • Google Workspace
  • Zoom
  • Salesforce
例え レストラン。 料理を作る必要はなく、食べるだけ。
IaaS・PaaS・SaaSの覚え方
覚える順番 

SaaS
完成品を使う

↓

PaaS
プログラムだけ作る

↓

IaaS
OSから自分で管理
種類 自分でやること
SaaS ほとんど何もしない
PaaS アプリ開発
IaaS OSから上を管理
覚え方 SaaSが一番楽。 IaaSが一番自由。 PaaSはその中間。
ミドルウェア（Middleware）（ミドルウェア）
一言でいうと
OSとアプリケーションの間で動き、 アプリが必要とする共通機能を提供するソフトウェアです。

なぜ必要？

アプリケーションは、通信やデータベース接続などを すべて自分で作るのは大変です。 そのため、共通機能をミドルウェアが担当します。

代表例
  • Tomcat：JavaのWebアプリを動かす
  • Apache HTTP Server：Webページを配信する
  • Nginx：Webサーバー・ロードバランサーとして利用
  • MySQL・PostgreSQL：データベース管理
イメージ 
アプリケーション
      │
ミドルウェア
(TomcatやMySQLなど)
      │
OS（Windows・Linux）
      │
ハードウェア
例え
レストランで考えると、
  • OS：建物
  • ミドルウェア：厨房・レジ・食器洗浄機
  • アプリ：料理人
料理人は厨房があるから料理に集中できます。
講師メモ
「Tomcatって何ですか？」 と聞かれたら、 「JavaのWebアプリを動かすミドルウェアです。」 と言えれば十分です。
ランタイム（Runtime）（ランタイム）
一言でいうと プログラムを実行するための環境。
例 JavaならJRE。 PythonならPython本体。 Node.jsもランタイムの一種。
「Javaを書いただけ」では動かず、 Javaを実行できる環境が必要です。
NIC冗長（Network Interface Card）（エヌアイシー）
一言でいうと ネットワークカードを複数用意して、 故障しても通信できるようにする仕組み。
会社の電話番号を2つ持つイメージ。 1つ故障してももう片方で連絡できる。
ハイパーバイザー（Hypervisor）（ハイパーバイザー）
一言でいうと 1台の物理サーバー上で、 複数の仮想サーバーを動かすためのソフト。
1軒のマンションの中に たくさんの部屋があるイメージ。 建物＝物理サーバー 部屋＝仮想マシン
仮想化基盤を操作するAPI（Application Programming Interface）（エーピーアイ）
一言でいうと
プログラムから仮想マシンやネットワークなどを 自動で操作するための窓口です。

APIとは？

APIは、ソフトウェア同士がやり取りするためのルールです。 人が画面をクリックしなくても、 プログラムから命令を送ることができます。

AWSの例

APIでできること
  • 仮想マシンを作る
  • 起動する
  • 停止する
  • 削除する
  • ネットワークを作る
  • ストレージを追加する
例え
レストランの注文票です。 店員（プログラム）が 「ラーメン1つ」 と注文票(API)を厨房へ渡すと、 料理人（クラウド）が料理を作ります。 厨房の中でどう作るかは気にしなくてよい、 というのがAPIの考え方です。
講師メモ
TerraformやAWS CLIなども、 最終的にはAWSのAPIを利用してクラウドを操作しています。
REST API（Representational State Transfer API）（レスト エーピーアイ）
一言でいうと
Webで最も広く使われているAPIの設計ルールです。

REST APIでは、 「何を操作したいか(URL)」 と 「何をしたいか(GETやPOSTなど)」 を組み合わせて通信します。

基本的な操作
メソッド 意味 例
GET 取得 社員一覧を見る
POST 登録 社員を登録する
PUT 更新 社員情報を変更する
DELETE 削除 社員を削除する
URL例 
GET    /employees
社員一覧を取得

GET    /employees/1001
社員1001を取得

POST   /employees
社員を登録

PUT    /employees/1001
社員1001を更新

DELETE /employees/1001
社員1001を削除
Spring Bootとの関係 
@GetMapping

@PostMapping

@PutMapping

@DeleteMapping
これらのアノテーションは REST APIの考え方に沿って作られています。
APIとの違い APIは「ソフト同士がやり取りする仕組み」の総称です。 REST APIは、そのAPIを作るときの代表的なルール（設計スタイル）の一つです。 つまり、
API
├ REST API
├ GraphQL
├ SOAP
└ RPC
という関係になります。
講師メモ
受講者が「RESTって何ですか？」と聞いてきたら、 「Web APIを作るときのお作法・ルールです」 と答えるとイメージしやすいです。
